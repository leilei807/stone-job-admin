## 一、背景与目标

- 项目 Fork 了开源 `xxl-job-admin`（**XXL-JOB 3.4.0**），其认证由外部 jar **XXL-SSO**（`com.xuxueli:xxl-sso-core`）承担：浏览器登录 `/auth/login` → 校验 `xxl_job_user` 表 → `XxlSsoHelper.loginWithCookie` 签发 token cookie。
- 诉求：将定时任务管理纳入 **Stone 统一体系**——Casdoor 单点登录 + Stone 既有 RBAC 权限，同时**保留可独立升级 xxl-job 的能力**（admin 侧对 Stone 工程零耦合）。
- 本期范围：**任务管理 + 日志查看 + 执行器管理 UI**；监控看板不做；**单租户**。

> **核心安全约束（最高优先级）**：管理平台一旦全部迁到 Stone，**admin 上的人类页面必须全部封锁、原生登录入口必须关闭**。否则 admin 就是一个绕过 Stone 权限体系（Casdoor 登录 + 角色分配的 `job:*` 菜单/接口权限）的后门——任何知道地址的人都能用原生账号直接登进后台。见第四章「无头封锁」。

> **重要修正（2026-07-21）**：clientToken **不是**静态密钥，而是 **Stone 用 Casdoor Client Credentials 换取的 JWT**（复用 Stone 现有 `common-security.ClientTokenProvider`）。admin 端用同一 Casdoor 公钥验签即可。

> **硬约束（2026-07-21，用户要求）**：**尽量新增文件，不改动原有文件。** 除 `pom.xml` 加第三方依赖声明外，所有业务代码/配置以新增文件落地，原有源码（`LoginController` / `XxlSsoConfig` / `SimpleLoginStore` / `application.properties` / `login.ftl`）一律不动。

## 二、为什么这样改（动机）

| 原痛点 | 无头化后的解法 |
|--------|----------------|
| admin 端人类 SSO 改造复杂：`CasdoorAuthFilter` / 角色白名单 / 登录页注入 | **全部取消**——人类不再登录 admin，这些都不需要了 |
| 角色在 Stone 的 `sys_role`，admin 读不到 → 权限闸门难做 | 闸门只放在 Stone，admin 完全不关心"谁有权限" |
| 原生 admin 账号（admin/123456）是越权后门 | 页面+登录入口封锁后，原生账号**物理上无法被使用** |
| `XxlJobAdminClient` 的 session-cookie hack（admin/123456 + 25min 续期） | 改为 Stone 用 clientToken 调 admin **机器端点**拿会话，密码不再硬编码 |

**结论**：无头化同时解决了"认证改造复杂"和"越权风险"两个问题。

## 三、目标架构

```
人类用户
   │ 仅在此登录（已有 Casdoor + 角色校验）
   ▼
┌─────────────────────────────────────────────┐
│  Stone 定时任务模块（唯一管理入口）            │
│  · 角色闸门：复用 Stone RBAC（角色分配 job:* 权限码即可）     │
│  · 管理 UI：任务 / 日志 / 执行器              │
│  · 机器凭证：clientTokenProvider 换 Casdoor JWT│
└───────────────────┬─────────────────────────┘
                     │  Authorization: Bearer <casdoorJwt>  （机器对机器）
                     ▼
┌─────────────────────────────────────────────┐
│  stone-job-admin（无头调度引擎）               │
│  · 人类页面：全部 403 封锁  ← 防越权核心        │
│  · 原生登录入口：关闭                          │
│  · 仅开放：/api/**（执行器协议）+ 机器会话端点  │
│  · 调度：JobScheduleHelper / TriggerPool / 注册 │
└─────────────────────────────────────────────┘
```

**一句话架构**：人类 → 只在 Stone 登录；Stone → 用 Casdoor clientToken 以机器身份调 admin 机器端点拿会话；admin → 页面封锁、只跑调度。

## 四、认证与安全设计（最关键一环）

### 4.1 现状（已探明，供参考）

XXL-JOB 3.x 内置 **XXL-SSO**（`com.xuxueli:xxl-sso-core`，**外部 jar，不可改源码**）：

| 组件 | 文件 | 作用 |
|------|------|------|
| `LoginController` | `framework/controller/LoginController.java` | `/auth/login`、`/auth/doLogin`（校验 `xxl_job_user`）、`/auth/logout`、`/auth/updatePwd` |
| `XxlSsoConfig` | `framework/web/xxlsso/XxlSsoConfig.java` | 注册 `XxlSsoWebInterceptor` 拦截 `/**`；`excludedPaths` 来自 `@Value("${xxl-sso.client.excluded.paths}")` |
| `SimpleLoginStore` | `framework/web/xxlsso/SimpleLoginStore.java` | 实现 `LoginStore`：token 签名存 `xxl_job_user.token` |
| `OpenApiController` | `business/scheduler/openapi/OpenApiController.java` | `/api/{uri}`，用 `XXL_JOB_ACCESS_TOKEN`+appname（**执行器通信**，保留不动） |

- 浏览器登录后拿 **XXL-SSO token（cookie）**，靠 cookie 通过 `XxlSsoWebInterceptor`。
- `LoginController` 的方法靠 `@XxlSso(login=false)` 注解放行匿名访问——这是关键：我们新增的机器端点同样可用此注解放行。
- 前端是 **内嵌 FreeMarker + jQuery**（`templates/framework/base/login.ftl`），**不是独立 Vue 工程**——无头化后整个前端都不再需要。

**关键结论**：认证闸口是 `XxlSsoWebInterceptor`（jar 内，不能改），但可在它**之前**用 Servlet Filter 拦截/放行，实现无头封锁与机器会话签发，XXL-SSO 源码零侵入。

### 4.2 认证方案演进：从「cookie 桥接」到「机器 token 前置注入」（2026-07-22 修订）

> 原（已废弃）方案：Stone 用 clientToken 调 `POST /auth/session/create` 换 SSO cookie，再带 cookie 调管理 API（即「cookie 桥接」）。该方案已删除，详见 §4.4。

现方案（无状态、更稳）：

- `clientToken` = Stone 用 Casdoor Client Credentials 换取的 RS256 JWT（`Bearer` 形式），由 `ClientTokenProvider` 缓存 / 轮换。
- admin 的 `XxlSsoWebInterceptor`（XXL-SSO，外部 jar 不可改）**只认 XXL-SSO cookie**——但我们已经把管理 API 路径 `/jobinfo/**`、`/joblog/**`、`/jobgroup/**` 加入了 `xxl-sso.client.excluded.paths`，使该拦截器对这些路径**直接放行**。
- 在这些路径上，由**新增的 `MachineAuthFilter`（`HIGHEST_PRECEDENCE + 1`，位于 `XxlSsoWebInterceptor` 之前）**拦截：取 `Authorization: Bearer`，用 Casdoor 公钥验签 → 按配置服务账号 `CasdoorProperties.serviceAccount` 查/建 `xxl_job_user` → 直接 `request.setAttribute("xxl_sso_user", LoginInfo)` 注入登录态。
- XXL-JOB 业务控制器用 `XxlSsoHelper.loginCheckWithAttr(request)` 读取该 request 属性（而非 SSO cookie）判定已登录。

**为什么不再需要 cookie 桥接**：登录态以「request 属性」而非「SSO cookie」传递，绕开了 XXL-SSO 的 cookie 校验；无会话缓存、无过期重换、无 `session/create` 验签失败故障点；且天然无状态，多实例部署无需共享会话。详见 §4.4。

### 4.3 无头封锁（防越权核心，全部新增文件）

> 这是本方案区别于"普通 SSO 改造"最关键的一章。目标：**让 admin 对人类完全不可达**，只保留机器可访问的极小接口面。

**封锁规则（`HeadlessAccessFilter`，最高优先级 Filter）**

| 请求 | 处置 | 说明 |
|------|------|------|
| `Accept` 含 `text/html` 的任意请求 | **403 拒绝** | 浏览器页面加载（首页 `/`、各管理页、登录页 `/auth/login` 等）——人类无法打开任何页面 |
| `/auth/login`、`/auth/doLogin`、`/auth/logout`、`/auth/updatePwd` | **403 拒绝** | 原生登录入口关闭，原生账号（admin/123456）**物理上无法登录** |
| `/api/**`（执行器协议） | **放行** | `OpenApiController`，用 `XXL_JOB_ACCESS_TOKEN` 校验，调度必需，保留 |
| `/actuator/**` | **放行** | 健康检查，运维需要 |
| 其它（含 `/jobinfo/*`、`/joblog/*`、`/jobgroup/*` 等管理 JSON API） | **放行** | 由 Stone 机器调用；人类拿不到凭证。这些路径已被加入 `xxl-sso.client.excluded.paths`，`XxlSsoWebInterceptor` 对其放行，改由 `MachineAuthFilter`（Bearer 验签）注入登录态，无需 SSO cookie |

**为什么能防越权**：人类浏览器请求是 `text/html` → 全 403；原生登录入口 → 全 403；管理 JSON API 虽放行，但调用它们必须携带 `Authorization: Bearer <clientToken>`（由 `MachineAuthFilter` 验签并注入登录态）——人类既无 clientToken（机器凭证，仅 Stone 后端持有）、也无登录入口，因此即便知道 API URL 也无法通过 `MachineAuthFilter` 的验签。后门被彻底堵死。

### 4.4 机器身份前置注入（`MachineAuthFilter`，替代已删除的 `CasdoorSessionController`）

> **变更（2026-07-22）**：原 `POST /auth/session/create`（`CasdoorSessionController`，cookie 桥接）已**删除**。改为 `MachineAuthFilter` 在每个管理请求上**无状态注入**登录态，不再签发 / 缓存 SSO cookie。

`MachineAuthFilter`（`framework/web/casdoor/MachineAuthFilter.java`，`@Component`，由 `CasdoorSecurityAutoConfiguration` 以 `HIGHEST_PRECEDENCE + 1` 注册，仅拦截 `/jobinfo/`、`/joblog/`、`/jobgroup/`）：

1. 取 `Authorization: Bearer <clientToken>`；缺失或格式错 → `401 {"code":401,"msg":...}`。
2. `CasdoorJwtVerifier.verifyAndGetSubject(jwt)` 用 Casdoor 公钥验签（校验 issuer/exp）；失败 → `401`。
3. `CasdoorUserResolver.resolve(properties.getServiceAccount())` 按**固定的服务账号**（`CasdoorProperties.serviceAccount`，如 `stone-job-system`）查/建 `xxl_job_user`。Stone 已在自身侧完成 `job:*` 权限校验，admin 只认"已授权机器"这一身份。
4. 构造 `LoginInfo` 并 `request.setAttribute("xxl_sso_user", loginInfo)`：
   - **必须用 7 参完整构造器**并显式 `roleList = ["ADMIN"]`。若误用 2 参构造器，仅设 `userId/signature`，`roleList` 为 `null`，会触发 `JobGroupPermissionUtil.hasRole` 失败 → `权限拦截[username=null]`（联调中实际踩坑，见 §16）。
   - 服务账号在 `CasdoorUserResolver` 中固定为管理员（`role=1`），admin 无需做细粒度判断。

`StoneJobReadController`（`business/controller/StoneJobReadController.java`，新增）的 `getJobInfo` / `getJobLog` 即依赖该注入的登录态做 `XxlSsoHelper.loginCheckWithAttr(request)` 断言。

**角色闸门不在 admin**：admin 只认 Stone 送来的、已授权过的机器身份；"该用户是否允许管理定时任务"由 **Stone 侧既有 RBAC 决定**——目标角色被分配 `job:*` 菜单/接口权限即可。admin 侧不持有、不判定角色。

### 4.5 clientToken 说明（非静态密钥）

- clientToken = **Stone 用 Casdoor Client Credentials 换取的 JWT**（短期有效、自动轮换）。Stone 已有 `ClientTokenProvider`（`common-security`），admin 端用同一把 Casdoor 公钥验签。
- admin 端只保管 **Casdoor 公钥**；私钥始终在 Casdoor。比静态密钥/原生账号安全得多。

## 五、核心架构决策

| 决策点 | 结论 |
|--------|------|
| admin 角色 | **无头化**：人类永不登录 admin，只跑调度 / 触发 / 心跳 |
| 管理平面 | **全部在 Stone**：任务 / 日志 / 执行器 UI 均属 Stone 定时任务模块 |
| 防越权 | admin 人类页面 + 原生登录入口**全 403 封锁**；原生 admin/123456 物理不可用 |
| 权限闸门 | **100% 由 Stone RBAC**（`job:*` 权限码），无 admin 侧角色名白名单 |
| 租户 | 本期**单租户** |
| admin 依赖 | 仅新增 `jjwt` 一个第三方 jar；**不引 common-security** |
| 原文件改动 | **零改动**（除 `pom.xml` 加依赖声明）；全部新增文件 |

**原文件「零改动」对照表**

| 原文件 | 是否改动 | 说明 |
|------|------|------|
| `LoginController.java` | ❌ 不改 | 其登录端点被 `HeadlessAccessFilter` 在 Filter 层 403，无需删代码 |
| `XxlSsoConfig.java` | ❌ 不改 | Filter 用 `FilterRegistrationBean` 独立注册 |
| `SimpleLoginStore.java` | ❌ 不改 | 账号查/建由 `CasdoorUserResolver` 负责 |
| `application.properties` | ⚠️ **已改（必要例外）** | 在 `xxl-sso.client.excluded.paths` 追加 `/jobinfo/**,/joblog/**,/jobgroup/**`，使 `XxlSsoWebInterceptor` 对这些管理 API 放行、改由 `MachineAuthFilter` 注入登录态。**这是「原文件零改动」原则的已知唯一例外**（详见 §15 决策点 7），因 `excluded.paths` 为单值逗号串、难以靠叠加 `@PropertySource` 安全追加 |

> **边界澄清**：`common-security` **仅引用于 Stone 主工程的 `stone-system-job` 模块**（`ClientTokenProvider` 所在地）。**stone-job-admin 侧不要引入 common-security**——会级联拉入 Shiro/common-core/common-redis，且其 `SecurityAutoConfiguration` 默认需 `classpath:cert/casdoor-public.key` 并装配 Shiro/Jwt/AOP bean，与 XXL-SSO 冲突、启动失败。admin 端仅依赖 jjwt，自写薄验签层，对 Stone 主工程零代码耦合。

## 十四、风险与提示

- **GPL-3.0**：Fork 在**私有仓库**、仅内部使用/修改，**不触发**源码分发义务。当前用法安全。
- **xxl-sso-core 黑盒**：无头封锁与机器会话都在它"外面"做（Servlet Filter + 新增 Controller），升级 xxl-job 时只需保留新增文件，核心 jar 零侵入、原文件零改动。

- **GLUE 模式**：admin 浏览器内 Groovy 编辑器极难迁移；本项目只用 BEAN handler（`@XxlJob`），直接跳过。

## 十六、定制 fork 接口偏差与联调修复记录（2026-07-22）

> 本 fork 在标准 xxl-job 3.4.0 基础上做了接口改名 / 字段改名 / 返回格式调整，且**没有**单条查询端点。Stone 后端 `XxlJobAdminClient` 初版直接套用标准 xxl-job 形态，联调中逐层暴露 5 类偏差，已全部修复。以下为权威对照，后续维护务必遵守。

### 16.1 端点重命名（标准 xxl-job → 本 fork）

| 操作 | 标准 xxl-job | 本 fork（正确值） | 客户端方法 |
|------|-------------|------------------|-----------|
| 新增任务 | `/jobinfo/add` | `/jobinfo/insert` | `callTaskAdd` |
| 新增执行器 | `/jobgroup/save` | `/jobgroup/insert` | `callGroupSave` |
| 删除任务 | `/jobinfo/remove` | `/jobinfo/delete` | `callTaskDelete` |
| 删除执行器 | `/jobgroup/remove` | `/jobgroup/delete` | `callGroupRemove` |
| 单条任务查询 | （无） | `POST /jobinfo/get`（新增 `StoneJobReadController`） | `callTaskGet` |
| 单条日志查询 | （无） | `POST /joblog/get`（新增 `StoneJobReadController`） | `callLogGet` |

> `update / start / stop / trigger / pageList / logKill` 路径与标准一致，未改名。

### 16.2 请求参数 key 偏差

| 场景 | 标准/直觉 key | 本 fork 实际 key |
|------|--------------|-----------------|
| 删除 / 启停 任务 / 执行器 | `id` | `ids[]`（数组，如 `params.put("ids[]", id)`） |
| 执行器名称 | `title` | `name`（admin 模型字段为 `name`） |

### 16.3 响应体字段偏差（致命）

- 标准 xxl-job 的 `Response<T>` 序列化字段为 **`content`**；本 fork 改为 **`data`**。
- `XxlJobAdminClient.doPostForContent` / `doPostForJson` 原读 `node.path("content")` → **改为 `node.path("data")`**（含 `convertValue` 取值），否则新增/查询返回的 id 或对象永远为 `null`/`0`，表现为"建任务成功但拿不到 xxlJobId"。

### 16.4 字段语义偏差：任务名 `jobDesc` ↔ `name`

- 本 fork 的 `XxlJobInfo` 模型字段名为 **`name`**（任务名称），而 Stone 侧领域模型 / DTO 用 `jobDesc`（任务描述）。
- 修复（仅 Stone 端，不动领域模型）：`infrastructure/dto/XxlJobInfo.jobDesc` 加 `@JsonProperty("name")`；`buildTaskParams` 由 `params.put("jobDesc", ...)` 改为 `params.put("name", info.getJobDesc())`。
- 漏改表现：`XXL-JOB error: 请输入任务名称`（admin `XxlJobServiceImpl.add` 校验 `getName()` 为空）。

### 16.5 登录态注入陷阱：`LoginInfo` 构造器

- `MachineAuthFilter` 注入 `xxl_sso_user` 时，**必须用 7 参完整构造器**并显式 `roleList = ["ADMIN"]`。
- 误用 2 参构造器（仅 `userId/signature`）→ `roleList=null` → `JobGroupPermissionUtil.hasRole(loginInfo,"ADMIN")` FAIL → `RuntimeException: 权限拦截[username=null]`（日志里的 `username` 实际源自 `roleList` 判定，并非真的 username 为空）。
- 修复：完整构造器 `new LoginInfo(userId, userName, realName, null, List.of(ADMIN_ROLE), null, expireTime, signature)`。

### 16.6 分页端点参数名偏差 + 编译器 `-parameters` 缺失（日志/分组分页崩溃）

**现象**：创建任务成功，但进入「调度日志」分页 `POST /joblog/pageList` 时 admin 抛：

```
java.lang.IllegalArgumentException: Name for argument of type [int] not specified,
and parameter name information not available via reflection.
Ensure that the compiler uses the '-parameters' flag.
```

前端表现为全局异常；`JobGroupController.pageList`（执行器/任务分组分页）同源，也会中招。

**根因 A（编译参数）**：admin 的 `JobLogController.pageList` / `JobGroupController.pageList` 用**裸 `@RequestParam`（不带 value）**绑定原始类型：

```java
public Response<...> pageList(HttpServletRequest request,
        @RequestParam(required=false, defaultValue="0") int offset,
        @RequestParam(required=false, defaultValue="10") int pagesize,
        @RequestParam int jobGroup, @RequestParam int jobId,
        @RequestParam int logStatus, @RequestParam String filterTime) { ... }
```

`@RequestParam` 无 value 时，Spring 必须靠 `-parameters` 编译参数从反射拿参数名；**未开 `-parameters` 即抛上述异常**。

**根因 B（profile 陷阱）**：根 POM 的 `<parameters>true</parameters>` 被包在一个 `<profile>` 内，且该 profile 的 `<modules>` 只含 `xxl-job-core`。**单独构建/运行 admin 模块**（如 `mvn -pl xxl-job-admin` 或直接 `spring-boot:run`）时该 profile 不生效 → admin 实际是**不带 `-parameters`** 编译的，触发崩溃。

**为什么建任务不崩、查日志崩**：`jobinfo/insert` 用 `XxlJobInfo` **命令对象**整体绑定（按字段名，不依赖参数名反射），故裸 `@RequestParam` 问题不暴露；`pageList` 用散装 `int` 参数，直接踩中。

**修复 A（构建配置，零源码改动）**：在 `xxl-job-admin/pom.xml` 的 `<build><plugins>` 中**无条件**新增 `maven-compiler-plugin` 并声明 `<parameters>true</parameters>`，不再依赖那个只含 `xxl-job-core` 的 profile。

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <parameters>true</parameters>
    </configuration>
</plugin>
```

**修复 B（客户端对齐 fork 契约）**：admin `pageList` 的分页参数名为 **`offset` / `pagesize`**（标准 xxl-job 用 `start` / `length`），分组名检索字段为 **`name`**（客户端此前发成 `title`）。在 `XxlJobAdminClient` 中修正：

- `callLogPage`：`params.put("start",...)` → `"offset"`、`params.put("length",...)` → `"pagesize"`。
- `callGroupPage`：同上 `offset`/`pagesize`，且 `params.put("title",...)` → `"name"`。

> 注：仅改发送键名，方法形参 `start`/`length`/`title` 保留不动；`JobAppService.pageLogs` 已算好的 `offset=(pageIndex-1)*pageSize` 原样传入即可。

### 16.7 时间字段类型错配：admin `Date` ↔ 客户端 `long`（日志分页反序列化崩溃）

**现象**：修复分页参数后，日志分页 `POST /joblog/pageList` 仍抛：

```
java.lang.IllegalArgumentException: Cannot deserialize value of type `long` from String "2026-07-22 15:27:42": not a valid `long` value
  ... XxlJobPageVO["data"] -> ArrayList[0] -> XxlJobLog["triggerTime"]
```

**根因**：admin `business/model/XxlJobLog` 中 `triggerTime` / `handleTime` 是 `java.util.Date`，被 admin 全局 Jackson 配置序列化为 **`"yyyy-MM-dd HH:mm:ss"` 字符串**（非时间戳）；而客户端反防腐层 DTO `infrastructure/dto/XxlJobLog` 把这两个字段声明成 `long`，`XxlJobAdminClient` 用 `new ObjectMapper()` 的 `convertValue` 反序列化时，无法把日期字符串解析成 `long` → 抛 `InvalidFormatException`。

**同根排查（任务侧无需改）**：admin `XxlJobInfo` 的 `triggerLastTime` / `triggerNextTime` 本就是 `long`（双方一致），`glueUpdatetime` / `addTime` / `updateTime` 为 `Date` 但客户端 DTO 已用 `String` 接收 —— 故任务详情单查 `callTaskGet` 不存在同类错配，无需改动。

**修复（仅 Stone 端，零改 admin 源码）**：

1. `infrastructure/dto/XxlJobLog.java`：`triggerTime` / `handleTime` 由 `long` 改为 `String`，对齐 admin 真实返回契约。
2. `app/JobAppService.java` 的 `toLogVO`：原用 `BeanUtils.copyProperties` 直接拷贝，类型不匹配（String→Long）会失败；改为**显式映射**，并用 `parseTime(String)` 把 `"yyyy-MM-dd HH:mm:ss"` 解析为 epoch 毫秒（按 `ZoneId.systemDefault()`）填入 `JobLogVO` 的 `Long` 字段。`getLog`（日志详情单查）同样走 `toLogVO`，一并受益。

> 设计原则重申：反防腐层 DTO 必须对齐 admin 的**真实返回契约**（字段名 + 字段类型），不能按标准 xxl-job 臆测。

### 16.9 分页响应体字段偏差：`PageModel.total` ↔ 客户端 `recordsTotal`/`recordsFiltered`（日志/分组分页反序列化崩溃）

**现象**：修复 16.6（分页参数名）后，日志分页 `POST /joblog/pageList` 仍抛：

```
java.lang.IllegalArgumentException: Unrecognized field "offset"
  (class com.stone.job.infrastructure.dto.XxlJobPageVO),
  not marked as ignorable (4 known properties: "recordsFiltered", "data", "page", "recordsTotal")
  ... XxlJobPageVO["offset"]
```

注意：请求体已正确发出（`offset/pagesize/...`），**这是响应反序列化失败，非请求问题**。

**根因**：admin 所有 `pageList` 返回 `Response<PageModel<T>>`。经 `javap` 反编译 `xxl-tool-2.5.0.jar` 实证，`PageModel` **只序列化 4 个字段**：`offset` / `pagesize` / `data`(List) / `total`。**完全没有**标准 xxl-job 的 `recordsTotal` / `recordsFiltered` / `page`。

而初版 `infrastructure/dto/XxlJobPageVO` 声明的是后者 → 响应里真实存在的 `offset/pagesize/total` 全是不认识字段，Jackson 先撞到 `offset` 直接抛 `UnrecognizedPropertyException`。

**更隐蔽的第二层 bug**：`JobAppService.pageLogs` 原读 `xxlResult.getRecordsFiltered()` 当分页总数，但 admin 压根不返回 `recordsFiltered` 而是 `total` —— 即使忽略未知字段不崩，分页总数也永远为 `0`。

**修复（仅 Stone 端，零改 admin）**：

1. `infrastructure/dto/XxlJobPageVO.java`：字段对齐 `PageModel`，声明为 `data` + `total` + `offset` + `pagesize`，并加 `@JsonIgnoreProperties(ignoreUnknown = true)` 防 fork 将来再漂移。
2. `app/JobAppService.java` 的 `pageLogs`：`getRecordsFiltered()` → **`getTotal()`**，分页总数才正确。

> 设计原则重申（与 16.7 同源）：反防腐层 DTO 必须对齐 admin 的**真实返回契约**，不能按标准 xxl-job 臆测。标准 xxl-job 用 `recordsTotal/recordsFiltered/page`，本 fork 用 `total/offset/pagesize`。

### 16.10 执行器管理「在线地址」全显示「离线」—— 两个相互独立、必须都修的坑

**界面现象**：Stone 前端「执行器管理」页，每条执行器「在线地址」列均显示「离线」（机器地址为空）。但 admin `/jobgroup/pageList` 直接用 token curl 返回 `stone-executor` 的 `registryList=["http://192.168.2.189:9999/"]`、`addressList` 也有值 → **admin 数据正常**，问题在 Stone 侧取在线状态。

**架构厘清**：Stone 前端「执行器管理」走 `JobGroupController.page → JobAppService.pageGroups`，**rows 读 Stone 本地 `job_group` 表**（`jobGroupGateway.page`，非直接转发 admin）；「在线地址」来自 `JobAppService.fetchOnlineAddresses()`，它内部 `callGroupPage(0,1000,"","")` 调 admin 取 `registryList`、按执行器标识填 `onlineAddresses`。

#### 坑 A：反防腐层 `XxlJobGroup` 反序列化异常被 `catch` 吞掉

- **根因**：`XxlJobAdminClient.MAPPER = new ObjectMapper()`（默认 `FAIL_ON_UNKNOWN_PROPERTIES=true`）。admin group 响应**多出 `accessToken` 与 `name` 字段**，而 `infrastructure/dto/XxlJobGroup` 只有 `title`（无 `name`）、无 `accessToken`，且未加 `@JsonIgnoreProperties(ignoreUnknown=true)` → `convertValue` 抛 `UnrecognizedPropertyException` → 被 `fetchOnlineAddresses` 的 `catch` 吞掉返回空 map → 所有组 `onlineAddresses` 为空。
- **修复**：`infrastructure/dto/XxlJobGroup.java` 加 `@JsonIgnoreProperties(ignoreUnknown = true)` + `title` 字段加 `@JsonProperty("name")`（对齐 admin 真实返回，手法同 16.4 的 `jobDesc↔name`）。

#### 坑 B：跨系统 `id` 漂移导致 `map.key` 对不上（修复 A 后仍空）

- **根因（用 curl 直接打 Stone 接口实锤）**：`fetchOnlineAddresses` 用 admin group 的 `id` 作 map 的 key，`pageGroups` 用本地 `xxlJobGroupId` 查 map。但**两侧 id 不一致**：
  - Stone 本地 `job_group` 表 `stone-executor`：`xxlJobGroupId = 1`
  - admin `/jobgroup/pageList` 返回的 `stone-executor`：`id = 3`
  - → `map.get(1)` 在 `{3: addr}` 里永远取不到 → `onlineAddresses` 仍空。
- **为什么 id 不一致**：admin 的 `xxl_job_group` 表被重建/数据迁移过，新插入的 `stone-executor` 自增 id=3；而 Stone 本地表里存的是更早创建时拿到的旧 id（或手工建 group 后 admin 侧又重置过）。
- **修复**：把匹配 key 从会漂移的 `id` 改为**稳定的 `appname`**（xxl-job 中 `appname` 是执行器稳定唯一标识，两侧一致都是 `stone-executor`）：
  - `fetchOnlineAddresses`：返回 `Map<String,String>`，`map.put(g.getAppname(), String.join(", ", g.getRegistryList()))`（原 `g.getId()`）。
  - `pageGroups`：`onlineMap.getOrDefault(vo.getAppname(), "")`（原 `vo.getXxlJobGroupId()`）。
  - 另加两行 `log.info`：打印在线地址 map 与每个 group 的匹配结果，便于重启后直接在控制台确认「是否正常返回」。
- **编译**：`mvn -pl stone-system-job/stone-system-job-app -am compile` ✅；重启 stone-system 后前端「在线地址」列显示 `http://192.168.2.189:9999/`。

> **经验沉淀（重要）**：跨系统按 id 关联时，若对端 id 可能漂移（重建/迁移/手工建），应优先用**业务稳定唯一键**（如 appname）而不是自增 id 关联。本坑 A、坑 B 相互独立——只修一个仍空，必须都修。

### 16.11 Stone 前端「执行器管理」两列语义：机器地址 vs 在线地址

前端列定义见 `web/stone-web/src/pages/job/group/index.tsx:120-128`。

| 列 | `dataIndex` | 数据来源 | 含义 |
|----|------------|---------|------|
| **机器地址** | `addressList` | Stone 本地 `job_group.address_list` | **手动配置/录入的地址列表** |
| **在线地址** | `onlineAddresses` | admin `xxl_job_group.registry_list`（执行器心跳实时上报） | **真实在跑的执行器地址** |

编辑表单 tooltip 原话（`group/index.tsx:221`）："注册方式选择「手动录入」时填写执行器机器地址，如 127.0.0.1:9999"。

**两种注册方式**（admin `xxl_job_group.address_type`）：

| 注册方式 | 机器地址 | 在线地址 | 适用场景 |
|---------|---------|---------|---------|
| **自动注册(0)** | 留空（显示 `-`） | 由执行器心跳自动入 `registry_list`，前端展示 | 弹性部署 / k8s 动态扩缩 / 机器会变 |
| **手动录入(1)** | **必填**（如 `http://192.168.2.189:9999`） | 由机器地址派生（基本一致） | 地址固定 / 内网隔离 / 不允许执行器主动注册 |

**结论**：当前 `stone-executor` 是**自动注册** → 机器地址 `-` 属预期，**「在线地址 `http://192.168.2.189:9999/`」才是真正在跑的执行器**，无需为"机器地址为空"担忧。**不建议**改成手动录入（会失去自动注册弹性，执行器重启/迁移都得改 Stone 配置）。
