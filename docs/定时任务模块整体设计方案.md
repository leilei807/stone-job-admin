## 一、背景与目标

- 项目 Fork 了开源 `xxl-job-admin`（**XXL-JOB 3.4.0**），其认证由外部 jar **XXL-SSO**（`com.xuxueli:xxl-sso-core`）承担：浏览器登录 `/auth/login` → 校验 `xxl_job_user` 表 → `XxlSsoHelper.loginWithCookie` 签发 token cookie。
- 诉求：将定时任务管理纳入 **Stone 统一体系**——Casdoor 单点登录 + Stone 既有 RBAC 权限，同时**保留可独立升级 xxl-job 的能力**（admin 侧对 Stone 工程零耦合）。
- 本期范围：**任务管理 + 日志查看 + 执行器管理 UI**；监控看板不做；**单租户**。

> **核心安全约束（最高优先级）**：管理平台一旦全部迁到 Stone，**admin 上的人类页面必须全部封锁、原生登录入口必须关闭**。否则 admin 就是一个绕过 Stone 权限体系（Casdoor 登录 + 角色分配的 `job:*` 菜单/接口权限）的后门——任何知道地址的人都能用原生账号直接登进后台。见第四章「无头封锁」。

> **重要修正（2026-07-21）**：clientToken **不是**静态密钥，而是 **Stone 用 Casdoor Client Credentials 换取的 JWT**（复用 Stone 现有 `common-security.ClientTokenProvider`）。admin 端用同一 Casdoor 公钥验签即可。

> **硬约束（2026-07-21，用户要求）**：**尽量新增文件，不改动原有文件。** 除 `pom.xml` 加第三方依赖声明外，所有业务代码/配置以新增文件落地，原有源码（`LoginController` / `XxlSsoConfig` / `SimpleLoginStore` / `application.properties` / `login.ftl`）一律不动。

## 二、为什么这样改（动机）

| 原痛点 | 无头化后的解法 |
|--------|----------------|
| admin 端人类 SSO 改造复杂：`CasdoorAuthFilter` / 角色白名单 / 登录页注入 | **全部取消**——人类不再登录 admin，这些都不需要了 |
| 角色在 Stone 的 `sys_role`，admin 读不到 → 权限闸门难做 | 闸门只放在 Stone，admin 完全不关心"谁有权限" |
| 原生 admin 账号（admin/123456）是越权后门 | 页面+登录入口封锁后，原生账号**物理上无法被使用** |
| `XxlJobAdminClient` 的 session-cookie hack（admin/123456 + 25min 续期） | 改为 Stone 用 clientToken 调 admin **机器端点**拿会话，密码不再硬编码 |

**结论**：无头化同时解决了"认证改造复杂"和"越权风险"两个问题。

## 三、目标架构

```
人类用户
   │ 仅在此登录（已有 Casdoor + 角色校验）
   ▼
┌─────────────────────────────────────────────┐
│  Stone 定时任务模块（唯一管理入口）            │
│  · 角色闸门：复用 Stone RBAC（角色分配 job:* 权限码即可）     │
│  · 管理 UI：任务 / 日志 / 执行器              │
│  · 机器凭证：clientTokenProvider 换 Casdoor JWT│
└───────────────────┬─────────────────────────┘
                     │  Authorization: Bearer <casdoorJwt>  （机器对机器）
                     ▼
┌─────────────────────────────────────────────┐
│  stone-job-admin（无头调度引擎）               │
│  · 人类页面：全部 403 封锁  ← 防越权核心        │
│  · 原生登录入口：关闭                          │
│  · 仅开放：/api/**（执行器协议）+ 机器会话端点  │
│  · 调度：JobScheduleHelper / TriggerPool / 注册 │
└─────────────────────────────────────────────┘
```

**一句话架构**：人类 → 只在 Stone 登录；Stone → 用 Casdoor clientToken 以机器身份调 admin 机器端点拿会话；admin → 页面封锁、只跑调度。

## 四、认证与安全设计（最关键一环）

### 4.1 现状（已探明，供参考）

XXL-JOB 3.x 内置 **XXL-SSO**（`com.xuxueli:xxl-sso-core`，**外部 jar，不可改源码**）：

| 组件 | 文件 | 作用 |
|------|------|------|
| `LoginController` | `framework/controller/LoginController.java` | `/auth/login`、`/auth/doLogin`（校验 `xxl_job_user`）、`/auth/logout`、`/auth/updatePwd` |
| `XxlSsoConfig` | `framework/web/xxlsso/XxlSsoConfig.java` | 注册 `XxlSsoWebInterceptor` 拦截 `/**`；`excludedPaths` 来自 `@Value("${xxl-sso.client.excluded.paths}")` |
| `SimpleLoginStore` | `framework/web/xxlsso/SimpleLoginStore.java` | 实现 `LoginStore`：token 签名存 `xxl_job_user.token` |
| `OpenApiController` | `business/scheduler/openapi/OpenApiController.java` | `/api/{uri}`，用 `XXL_JOB_ACCESS_TOKEN`+appname（**执行器通信**，保留不动） |

- 浏览器登录后拿 **XXL-SSO token（cookie）**，靠 cookie 通过 `XxlSsoWebInterceptor`。
- `LoginController` 的方法靠 `@XxlSso(login=false)` 注解放行匿名访问——这是关键：我们新增的机器端点同样可用此注解放行。
- 前端是 **内嵌 FreeMarker + jQuery**（`templates/framework/base/login.ftl`），**不是独立 Vue 工程**——无头化后整个前端都不再需要。

**关键结论**：认证闸口是 `XxlSsoWebInterceptor`（jar 内，不能改），但可在它**之前**用 Servlet Filter 拦截/放行，实现无头封锁与机器会话签发，XXL-SSO 源码零侵入。

### 4.2 为什么必须「cookie 桥接」而不是直连

- `clientToken` = Casdoor 签发的 RS256 JWT（`Bearer` 形式）。
- admin 内部认证闸口是 `XxlSsoWebInterceptor`（XXL-SSO，外部 jar 不可改），**只认 XXL-SSO 自己签发的 SSO cookie**。
- 直接拿 clientToken 当 Bearer 打 admin 管理 API → XXL-SSO 拦截器因无有效 cookie 拒绝。
- 且 XXL-JOB 业务代码从 SSO 会话取当前 `XxlJobUser`（操作人），无会话会 NPE。
- 因此必须在 XXL-SSO **之前**用机器端点把 clientToken 翻译成 SSO cookie。

### 4.3 无头封锁（防越权核心，全部新增文件）

> 这是本方案区别于"普通 SSO 改造"最关键的一章。目标：**让 admin 对人类完全不可达**，只保留机器可访问的极小接口面。

**封锁规则（`HeadlessAccessFilter`，最高优先级 Filter）**

| 请求 | 处置 | 说明 |
|------|------|------|
| `Accept` 含 `text/html` 的任意请求 | **403 拒绝** | 浏览器页面加载（首页 `/`、各管理页、登录页 `/auth/login` 等）——人类无法打开任何页面 |
| `/auth/login`、`/auth/doLogin`、`/auth/logout`、`/auth/updatePwd` | **403 拒绝** | 原生登录入口关闭，原生账号（admin/123456）**物理上无法登录** |
| `/api/**`（执行器协议） | **放行** | `OpenApiController`，用 `XXL_JOB_ACCESS_TOKEN` 校验，调度必需，保留 |
| `/actuator/**` | **放行** | 健康检查，运维需要 |
| `/auth/session/create`（机器端点） | **放行（但仅接受 clientToken）** | Stone 后端专用，见 4.4 |
| 其它（含 `/jobinfo/*`、`/joblog/*` 等管理 JSON API） | **放行** | 由 Stone 机器调用；人类拿不到会话（登录入口已封），即便直调 URL 也无凭证 |

**为什么能防越权**：人类浏览器请求是 `text/html` → 全 403；原生登录入口 → 全 403；管理 JSON API 虽放行，但调用它们需要 SSO 会话 cookie，而会话**只能**由 `/auth/session/create`（持 clientToken）签发——人类既无 clientToken、也无登录入口，因此即便知道 API URL 也无法获得任何会话。后门被彻底堵死。

### 4.4 机器会话端点（`CasdoorSessionController`，仅机器可调）

Stone 后端用 clientToken 向 admin 申请一次会话，供后续管理 API 使用：

- 新增 `POST /auth/session/create?casdoorUserId=xxx`
- 方法标 `@XxlSso(login=false)`（jar 提供，标注在自己方法上即放行，免改 `XxlSsoConfig`）；且仅接受 `Authorization: Bearer <casdoorJwt>`。
- 处理流程：`CasdoorJwtVerifier` 用 Casdoor 公钥验签 clientToken（校验 issuer/exp）→ 提取身份 → `CasdoorUserResolver` 按 `casdoorUserId` 查/建 `xxl_job_user`（用 `XxlJobUserMapper`，不改 `SimpleLoginStore`）→ 复刻 `LoginController.doLogin` 的 `XxlSsoHelper.loginWithCookie(...)` 签发 SSO cookie 返回。
- **角色闸门不在 admin**：admin 只认 Stone 送来的、已授权过的 `casdoorUserId`；"该用户是否允许管理定时任务"由 **Stone 侧既有 RBAC 决定**——目标角色被分配 `job:*` 菜单/接口权限即可。admin 侧不持有、不判定角色。

### 4.5 clientToken 说明（非静态密钥）

- clientToken = **Stone 用 Casdoor Client Credentials 换取的 JWT**（短期有效、自动轮换）。Stone 已有 `ClientTokenProvider`（`common-security`），admin 端用同一把 Casdoor 公钥验签。
- admin 端只保管 **Casdoor 公钥**；私钥始终在 Casdoor。比静态密钥/原生账号安全得多。

## 五、核心架构决策

| 决策点 | 结论 |
|--------|------|
| admin 角色 | **无头化**：人类永不登录 admin，只跑调度 / 触发 / 心跳 |
| 管理平面 | **全部在 Stone**：任务 / 日志 / 执行器 UI 均属 Stone 定时任务模块 |
| 防越权 | admin 人类页面 + 原生登录入口**全 403 封锁**；原生 admin/123456 物理不可用 |
| 权限闸门 | **100% 由 Stone RBAC**（`job:*` 权限码），无 admin 侧角色名白名单 |
| 租户 | 本期**单租户** |
| admin 依赖 | 仅新增 `jjwt` 一个第三方 jar；**不引 common-security** |
| 原文件改动 | **零改动**（除 `pom.xml` 加依赖声明）；全部新增文件 |

**原文件「零改动」对照表**

| 原文件 | 是否改动 | 说明 |
|------|------|------|
| `LoginController.java` | ❌ 不改 | 其登录端点被 `HeadlessAccessFilter` 在 Filter 层 403，无需删代码 |
| `XxlSsoConfig.java` | ❌ 不改 | Filter 用 `FilterRegistrationBean` 独立注册 |
| `SimpleLoginStore.java` | ❌ 不改 | 账号查/建由 `CasdoorUserResolver` 负责 |
| `application.properties` | ❌ 不改 | 配置走新增 `casdoor.properties` + `@PropertySource`；放行靠 `@XxlSso(login=false)` |
| `login.ftl` 等所有模板 | ❌ 不改 | 无头化后前端整体不再对人类可达，无需改动 |

> **边界澄清**：`common-security` **仅引用于 Stone 主工程的 `stone-system-job` 模块**（`ClientTokenProvider` 所在地）。**stone-job-admin 侧不要引入 common-security**——会级联拉入 Shiro/common-core/common-redis，且其 `SecurityAutoConfiguration` 默认需 `classpath:cert/casdoor-public.key` 并装配 Shiro/Jwt/AOP bean，与 XXL-SSO 冲突、启动失败。admin 端仅依赖 jjwt，自写薄验签层，对 Stone 主工程零代码耦合。

## 十二、取消的工作（重要）

以下原 SSO 改造设想**全部取消**，因为人类不再登录 admin：

- ❌ admin 端 Casdoor 人类登录（`CasdoorAuthFilter` 的人类分支 / 登录页按钮 / 回调）
- ❌ admin 端角色白名单（角色在 Stone，闸门只在 Stone）
- ❌ 改 `XxlSsoConfig` / `LoginController` / `SimpleLoginStore` / `application.properties` / `login.ftl`
- ❌ 前端 `casdoor-auth.js` 严格 Bearer 封装（不再需要）
- ❌ 监控看板（运行数/成功率/在线执行器/趋势）——本期不做

新增文件从「多套 Filter/Controller/前端注入」精简为：**`HeadlessAccessFilter` + `CasdoorSessionController` + `CasdoorUserResolver` + `CasdoorJwtVerifier` + `CasdoorProperties` + `CasdoorSecurityAutoConfiguration`**，全部机器侧，零人类交互。

## 十四、风险与提示

- **GPL-3.0**：Fork 在**私有仓库**、仅内部使用/修改，**不触发**源码分发义务。当前用法安全。
- **xxl-sso-core 黑盒**：无头封锁与机器会话都在它"外面"做（Servlet Filter + 新增 Controller），升级 xxl-job 时只需保留新增文件，核心 jar 零侵入、原文件零改动。
- **admin 必须继续运行**：无头化搬的是"控制台"，不是"引擎"。调度、触发、执行器心跳都在 admin 进程，需保持部署。
- **clientToken 安全**：Casdoor 签发短期 JWT、自动轮换；admin 仅保管公钥。
- **GLUE 模式**：admin 浏览器内 Groovy 编辑器极难迁移；本项目只用 BEAN handler（`@XxlJob`），直接跳过。

## 十五、决策点（已全部定稿）

1. **无需特殊角色名白名单（已取消此待确认项）**：复用 Stone 既有 RBAC——给目标角色分配 `job:*` 菜单/接口权限码即可。定时任务模块开发时已在 Flyway 建好 `job:*` 权限码与菜单/按钮，并接入 `PermissionGate` + `@RequiresPermissions`，闸门天然存在。
2. **本期 Stone 侧范围（已定稿）**：「任务管理 + 日志查看 + **执行器管理 UI**」。执行器管理 UI 经用户确认由「本期不做」修订为「做」，并已落地（见 7.3）。监控看板本期不做。
3. **`pom.xml` 加 `jjwt` 依赖**：✅ 已确认可接受（仅 `io.jsonwebtoken:jjwt` 一个第三方 jar 声明，admin 端用于验签 Casdoor JWT）。
4. **租户隔离**：✅ 本期仅做**单租户**，租户隔离列为后续阶段（见阶段 4）。
5. **clientToken 来源**：✅ 从 Casdoor Client Credentials 换取 JWT，非静态密钥；admin 用同一公钥验签。
6. **尽量新增文件不改动原码**：✅ 已执行，admin 端 6 个 Java 类 + 2 资源文件全新增，原文件零改动。
